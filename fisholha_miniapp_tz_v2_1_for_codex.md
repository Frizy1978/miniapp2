# ТЗ для Codex — Fish Olha Order Mini App v2.1

## 0. Контекст и цель проекта

Есть действующий сайт магазина рыбы и морепродуктов **fisholha.ru** на **WordPress + WooCommerce**.  
Он остается основным источником актуального каталога товаров.

Несколько раз в месяц магазин делает выездную поставку продукции в удаленные населенные пункты/по области.  
Нужно создать **Telegram Mini App** для покупателей, чтобы они могли **оформлять предзаказы** на ближайшую поставку.

### Ключевые особенности бизнеса
- **Онлайн-оплаты нет**
- Оплата производится **наличными при передаче заказа покупателю**
- Каталог и цены берутся из **WooCommerce**
- Заказы группируются по **приемам заказов / сессиям сбора**
- Админу нужен **собственный интерфейс**, где он:
  - запускает и закрывает прием заказов
  - видит заказы по каждому покупателю
  - видит сводную потребность по товарам
  - может экспортировать данные в **PDF**
- **Google Sheets не используются**

- собственная база приложения (PostgreSQL)

# 2. Что должно получиться в итоге

Нужно создать систему из 3 частей:

## 2.1 Telegram Bot
Нужен для:

- анализировать историю
- экспортировать документы в PDF

# 3. Роли и авторизация

## 3.1 Покупатель
### Авторизация:
Покупатель авторизуется **через Telegram Mini App auth**.

Используются данные Telegram:
- telegramUserId
- username
- first_name
- last_name

### Почему это правильно
Это естественный и надежный способ для Mini App.  
Не нужно отдельной регистрации и пароля.

---

## 3.2 Админ

### Принятое решение
Сделать **админский интерфейс скрытой отдельной зоной в этом же проекте**, но:
- не показывать покупателю никаких админских ссылок
- не добавлять покупателю пункты меню для админки
- проверять доступ только по роли `admin`

### Рекомендуемый вариант реализации
**Не делать отдельный логин/пароль для MVP.**  
Сделать:
- Telegram auth для всех
- роль `buyer` / `admin`
- allowlist по `telegramUserId` для админа
- отдельный route, например:
  - `/admin`
- доступ к нему только для пользователей с ролью `admin`

### Почему это лучший вариант
- покупатель вообще не видит админку
- не нужен отдельный auth flow
- проще и быстрее реализовать
- удобно в MVP
- при желании позже можно добавить отдельный admin login

### Важно
Покупателю не должны отображаться:
- ссылки на `/admin`
- пункты меню админа
- упоминания о наличии админ-панели

# 4. Основная бизнес-сущность — Прием заказов

Прием заказов — это отдельная сущность, которая объединяет все заказы на конкретный выезд/поставку.

## 4.1 Название сущности
Рекомендуемое название:
- `OrderBatch`

## 4.2 Что хранит прием заказов
- id
- порядковый номер приема заказов
- дата и время старта
- дата и время окончания приема заказов
- дата и время поставки/выдачи
- статус:
  - draft
  - open
  - closed
  - archived
- комментарий/описание для покупателей
- createdBy admin
- createdAt
- updatedAt

## 4.3 Логика
- Пока есть активный batch со статусом `open`, покупатели могут делать и редактировать заказы
- Когда время окончания прошло или админ вручную закрыл прием — новые заказы и редактирование запрещаются
- На главной странице Mini App должно отображаться:
  - **Открыт прием заказов**
  - или **Прием заказов завершен**

---

# 5. Формат ID приема заказов и заказа

## 5.1 Batch ID
Формат:
- `26-001`

где:
- `26` — последние 2 цифры года
- `001` — последовательный номер приема заказов в рамках года

## 5.2 Order ID
Формат:
- `26-001-0001`

где:
- `26-001` — batch
- `0001` — последовательный номер заказа в рамках batch

# 6. Сценарий работы системы

## 6.1 Запуск приема заказов
1. Админ запускает новый прием заказов
2. Указывает:
   - дату и время окончания приема
   - дату и время поставки
   - при необходимости текстовое сообщение/описание
3. Бот рассылает или публикует ссылку/кнопку Mini App в Telegram-группе покупателей

## 6.2 Вход покупателя
1. Покупатель открывает бота
2. Нажимает кнопку открытия Mini App
3. Авторизуется через Telegram автоматически
4. Попадает на главную страницу

## 6.3 Оформление заказа
1. Покупатель открывает каталог
2. Ищет товар по поиску или категориям
3. Добавляет позиции в корзину
4. Указывает количество / вес
5. Проверяет заказ
6. Подтверждает заказ
7. Заказ сохраняется в базу

## 6.4 Редактирование заказа
Пока batch открыт:
- покупатель может открыть свой заказ
- скорректировать состав
- изменить вес/количество
- повторно подтвердить изменения

После закрытия batch:
- редактирование запрещено

## 6.5 После подтверждения
- Админ получает уведомление в Telegram:
  - ID заказа
  - покупатель
  - состав заказа
  - сумма

## 6.6 После завершения приема
- Покупатели видят на главной:
  - **Прием заказов завершен**
- Добавление новых заказов запрещено
- Редактирование существующих заказов запрещено

# 7. Функционал покупателя

## 7.1 Главная страница (Main)
На главной странице должны быть 2 динамических блока:

### Блок 1
- статус приема:
  - **Открыт прием заказов**
  - или **Прием заказов завершен**
- дата и время окончания приема
- статический текст:
  - **Оплата производится при получении заказа**

### Блок 2
- дата и время поставки/выдачи
- без указания адреса

### Ниже
- кнопка **Открыть каталог**

## 7.2 Каталог
Форма каталога должна включать:
- header:
  - логотип справа
  - иконка возврата на Main
- поле поиска по всему ассортименту
- карусель категорий
- раздел **Новинки**
  - карусель 2x2
  - максимум 8 товаров
- нижнее меню:
  - Главная
  - Каталог
  - Корзина
  - Профиль

## 7.3 Категория
При выборе категории открывается форма с товарами данной категории.

## 7.4 Корзина
Покупатель может:
- добавлять товары
- удалять товары
- менять количество
- указывать вес / количество
- перейти к подтверждению

## 7.5 Подтверждение заказа
На экране подтверждения должны быть:
- перечень товаров
- количество / вес
- цена по позиции
- общая сумма
- кнопка подтвердить
- кнопка вернуться к редактированию

## 7.6 Профиль покупателя
Минимально:
- Telegram данные
- история своих заказов
- детали прошлых заказов

# 8. Функционал администратора

## 8.1 Управление приемом заказов
Админ может:
- создать новый прием заказов
- открыть прием заказов
- продлить дату и время окончания
- закрыть прием вручную
- указать дату и время поставки

## 8.2 Просмотр заказов
Админ должен иметь 2 основных режима просмотра:

### Режим 1 — по покупателям
Показывать:
- список покупателей
- по каждому покупателю:
  - ID заказа
  - дата
  - состав заказа
  - количество / вес
  - сумма
  - статус заказа
- возможность открыть подробную карточку заказа

### Режим 2 — сводка по товарам
Показывать:
- все товары в рамках конкретного приема заказов
- общий объем по каждой позиции:
  - количество
  - вес
- число покупателей, заказавших товар
- ориентировочная сумма по позиции
- это нужно для анализа:
  - что есть в наличии
  - что нужно докупить

## 8.3 Экспорт в PDF
Нужно сделать минимум 2 типа PDF:

### PDF 1 — заказы по покупателям
- группировка по покупателям
- у каждого покупателя:
  - номер заказа
  - состав заказа
  - количество / вес
  - сумма

### PDF 2 — сводная ведомость по товарам
- товар
- общий объем
- единица измерения
- количество покупателей
- ориентировочная стоимость

### Требования
PDF должен быть пригоден для печати.

## 8.4 История приемов заказов
Админ должен видеть список предыдущих batch:
- дата начала
- дата окончания
- дата поставки
- количество заказов
- сумма всех заказов
- статус

## 8.5 Аналитика для владельца
Минимально:
- количество заказов по batch
- средний чек по batch
- средний чек по покупателю
- наиболее популярные товары
- наиболее популярные категории
- количество уникальных покупателей в batch
- общая выручка по batch (ориентировочная)

## 9.1 Источник данных

## 9.2 Какие поля использовать из WooCommerce

- категории
- название товара
- изображение товара
- цену
- slug
- SKU, если есть
- featured/new flags, если есть
- `_price_unit` как единицу измерения цены

## 9.3 Важное ограничение
Mini App **не управляет каталогом**.  
Каталогом управляют только через WooCommerce.

# 10. Поиск

Поиск должен работать:
- по части названия товара
- без необходимости вводить полное имя
- без учета регистра

Например:
если товар называется:
- `Горбуша в собственном соку`

поиск должен находить по:
- `горб`
- `собств`
- `соку`

# 11. Работа с весовыми товарами

## Принятое решение
Для весовых товаров покупатель **вводит предполагаемый вес вручную**.

### Что нужно поддержать
- дробные значения
- понятную единицу измерения
- валидацию положительного значения

### UX
Для весового товара:
- вместо количества показывать поле веса
- единица должна подставляться из `_price_unit`, если применимо

# 12. Статусы заказа

## Для MVP
Нужны только:
- `created`
- `accepted`

### Дополнительно
Когда покупатель вносит изменения в уже существующий заказ в открытом batch, это считается:
- корректировкой существующего заказа
- а не созданием нового order record, если не выбрана другая бизнес-модель

### Рекомендуемая логика
- заказ создается один раз
- потом обновляется
- статус может оставаться `created`
- админ вручную может перевести заказ в `accepted`

# 13. География

## Принятое решение
Система не привязана к одному городу.  
Нужно проектировать ее как систему для **удаленной доставки по области / нескольким населенным пунктам** в будущем.

### Для MVP
Можно не делать сложную географию, но:
- не зашивать город в архитектуру
- предусмотреть возможность хранить у покупателя:
  - город / населенный пункт
  - адрес / заметку по доставке, если понадобится позже
